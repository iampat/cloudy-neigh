Storage abstraction · design note

Atomic writes, CAS, and atomic rename over one blob interface

Local POSIX, GCS, and S3 — behind a single interface that guarantees the four semantics gocloud.dev/blob won't give you portably. The trick: never map your layout onto storage keys. Own the layout, and every operation collapses to two things all three backends can do.

The one idea

Five requirements reduce to two primitives

Store two kinds of thing differently. Data is written once and never changed. One small pointer — the root — says which data is current, and it's the only thing ever mutated, always by compare-and-swap.

With that split, atomic write, write-if-not-exist, CAS, multi-object rename, and prefix rename all become logic over exactly two storage capabilities:

Capability 1

Put(key, data)

Atomically write a new, uniquely-named, immutable object. Native on all three backends — nothing to build.

Capability 2

CAS(root, expect, data)

Swap one object only if its version still matches. GCS generation · S3 If-Match · local flock. The entire non-portable surface.

immutable data CAS'd root manifest entry reference garbage (old version)
Architecture

Two layers — only the bottom one knows the backend

The logical layer is written once and is fully portable. Underneath it, a small Store gives three implementations of the same five methods; the only genuinely backend-specific method is CAS.

Read as: app → portable logic → one tiny per-backend primitive → gocloud.dev/blob. Swapping backends touches only the three green boxes.
Data model

What actually sits in the bucket

Your logical paths never become storage keys. Data objects get opaque, unique names and are written exactly once. The root is a single object holding the manifest: logical path → data object. Because the root carries a version, it is the one point where concurrency is resolved.

Invariant: data objects are content-addressed / uuid-named and never rewritten — so CAS is needed on the root only. Everything else is a plain atomic Put.
group A

Single-object operations

Each is: write the new data as a fresh immutable object, then move the pointer with one CAS. The CAS is the atomic instant — readers see the old object or the new one, never a torn state, never nothing.

Atomic write

Write(path, data) → Handle
  1. Put("blobs/<new>", data) — a brand-new immutable object; can't clash with anything.
  2. Read root (version v7); build the manifest with path → B2.
  3. CAS(root, v7, manifest') succeeds → v8. On ErrConflict, someone else moved first: re-read and retry.

Write-if-not-exist

WriteIfNotExist(path, data)
  1. Read root; if path already present → return ErrExists immediately.
  2. Else build manifest with the new entry and CAS(root, v, …).
  3. If the CAS loses the race, retry: the re-read now shows the path present → ErrExists. Exactly one writer creates it.

At the primitive level this is Store.CAS(key, expect=∅, data) — GCS DoesNotExist, S3 If-None-Match:*, local O_EXCL. The library's portable IfNotExist works for the two clouds but is racy on local, so the local Store uses O_EXCL/flock instead.

Compare-and-swap

CompareAndSwap(path, expect Handle, data) → Handle
  1. Read root; look up the current handle for path.
  2. If it differs from the caller's expect → ErrConflict (someone changed it since the caller last read).
  3. Else Put the new blob and CAS(root). The root's own version also guards against a concurrent structural edit slipping in between.
group B

Structural operations

This is the payoff. Because logical layout lives inside the manifest, moving objects is editing the manifest — the data never moves. Any number of moves land together under a single root CAS, so they're atomic by construction.

Atomic multi-object rename

RenameMany([a→x, b→y])
  1. Read root; build a new manifest with a→x and b→y relabelled (entries still point at the same B1/B2).
  2. A single CAS(root, v7, manifest') commits all moves at once. N moves · 0 bytes copied · 1 CAS.

Atomic prefix / folder rename

Rename("dir/", "arch/")
  1. In the manifest, re-point the whole dir/ subtree to sit under arch/ — a key change on one node, not a per-file copy.
  2. Commit with a single CAS(root). The prefix appears entirely renamed or not at all; no reader ever sees a half-renamed folder.

Object stores have no folders to rename — but the manifest does, and swapping it is one atomic op. This is why the requirement is satisfiable even though S3 has no native rename at all.

Scaling

When one manifest object gets too big

A single root works to thousands of entries. Beyond that, shard the manifest into a tree of CAS-able nodes and use copy-on-write: a change rewrites only the path from the touched node up to the root, then does one root CAS. Still atomic, still no data movement — now O(depth) small writes instead of one large one.

COW: only the ancestors of the change are rewritten (amber). Untouched subtrees (node B, leaf B1) are shared by reference. Old root + node A become garbage after the CAS.
What you take on

The honest costs

Garbage collection

Superseded objects must be swept

Old data + manifest nodes are unreferenced after a CAS. Mark-from-root sweep with a grace period for in-flight readers. Real code you must write.

Read indirection

Reads resolve through the root

Cache the manifest and revalidate cheaply via the root's version (generation / HEAD ETag / stat) — otherwise +1 round trip per read.

Write contention

Structural writes serialize on the root

High concurrency → CAS retries. Shard the root per top-level namespace to spread the contention across independent roots.

Backend limits

Two edges to verify

Local CAS is single-host only (flock ≠ NFS-safe). S3 If-Match is real on AWS but must be confirmed on S3-compatible endpoints.
